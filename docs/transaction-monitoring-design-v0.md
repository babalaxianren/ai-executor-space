## 一、背景与问题

### 当前痛点

**业务侧：**
- 无法直观看到用户在交易漏斗哪一步流失
- 无法判断流失是用户主动放弃还是技术故障导致
- 支付渠道的表现数据分散在各第三方平台，无法横向对比

**技术侧：**
- 错误监控分散在 Sentry 和 Datadog，两者无法联动
- 无法看到 API/SDK 错误导致的交易失败率
- 告警响应不够灵敏，且缺乏业务影响的量化

**运营侧：**
- 各市场支付账号的健康状态分散在不同区域负责人手中
- 无统一入口判断支付账号是否出现异常

### 本次范围

- 渠道：**线上（Online）**
- 市场：覆盖当前所有已上线市场，支持新市场扩展
- 版本：v1 不包含风险合规层（Layer 5），后续视需求接入

## 二、系统目标

### 核心目标

> 让工程、PM 和电商领导能在同一个系统里，回答各自最关键的问题。

### 每类用户的核心问题

| 用户 | 核心问题 | 使用频率 |
|---|---|---|
| 工程师 | 现在有没有问题？出了问题在哪里？ | 实时，出问题才看 |
| PM | 转化率为什么下降？哪个步骤需要优化？ | 每周，做优化决策 |
| 电商领导 | 哪个市场表现最好？支付策略是否需要调整？ | 每月，做资源分配 |

### 成功的标准（待填写）

- [ ] 工程师打开系统，5 秒内能判断当前交易是否健康
- [ ] PM 能看到分市场、分步骤的漏斗转化，并能关联到技术失败原因
- [ ] 电商领导能看到各支付渠道的成功率和用户参与度对比
- [ ] 某市场支付账号成功率异常下降时，系统自动触发告警

## 三、核心数据模型

### Transaction ID

**问题：** 目前没有统一的 Transaction ID 在前后端流转，导致前端行为数据、API 日志、支付 SDK 回调三者无法关联，出现问题时只能分别排查，无法还原一笔交易的完整链路。

**设计决策：前端生成，后端确认**

- 前端在用户进入结算流程时生成 `transaction_id`（UUID）
- `transaction_id` 随所有后续请求传递（API Header / Event 属性）
- 后端在收到第一个请求时记录并绑定，之后以后端记录为准
- 所有埋点事件、API 日志、支付 SDK 回调均携带此 ID

**格式：** `txn_{timestamp}_{random}` （待与后端确认）

**为什么前端生成，而不是后端生成？**

后端生成 ID 的问题在于：后端只有在收到请求后才能产生 ID，这意味着用户在发出任何请求之前的行为（浏览支付方式、在地址页停留多久、中途切换渠道后放弃）全部丢失。这部分数据恰恰是分析"用户为什么没有完成支付"最关键的信息。

前端生成可以从用户进入结算流程的第一秒开始追踪，漏斗数据完整，包括"进入了结算但从未发起任何请求就离开"的用户，这类用户在现有数据中是完全不可见的。

**为什么不只由前端生成，而是需要后端确认？**

纯前端生成有可靠性风险：页面刷新、网络重试、多标签页操作可能产生重复或孤立的 ID，导致同一笔交易被计算多次。后端确认的作用是：在第一个 API 请求到达时，将前端 ID 与后端的订单记录绑定，之后以后端为权威数据源。前端负责"开始追踪"，后端负责"保证准确"。

**这个方案的整体优势：**

| 对比项 | 纯后端生成 | 纯前端生成 | 前端生成+后端确认 |
|---|---|---|---|
| 漏斗数据完整性 | 缺失第一步之前的行为 | 完整 | 完整 |
| 数据可靠性 | 高 | 有重复/丢失风险 | 高 |
| 实现复杂度 | 低 | 低 | 中 |
| 适合本系统 | 否 | 否 | 是 |

### 交易生命周期状态机

```
initiated          用户进入结算流程
  ↓
address_entered    填写地址完成
  ↓
shipping_selected  选择配送方式
  ↓
payment_selected   选择支付方式
  ↓
payment_submitted  提交支付
  ↓
payment_processing 支付处理中（等待第三方响应）
  ↓
completed          支付成功
failed             支付失败（技术原因）
abandoned          用户主动放弃
```

> 待补充：各市场的步骤是否一致？线下 POS 流程（本版本不覆盖）

### 交易失败分类

| 失败类型 | 定义 | 归因 |
|---|---|---|
| user_abandoned | 用户停留超过 X 分钟无操作后离开 | 用户行为 |
| api_timeout | 后端 API 响应超时 | 技术-后端 |
| sdk_error | 支付 SDK 返回错误码 | 技术-第三方 |
| payment_declined | 支付被发卡行拒绝 | 用户/支付 |
| validation_error | 前端表单校验失败 | 用户行为 |
| network_error | 前端网络请求失败 | 技术-前端 |

## 四、功能模块

### Module 1：系统健康总览（Health Overview）

**主要用户：** 工程师

**页面功能：**
- 当前状态指示：健康 / 警告 / 故障（基于各市场实时指标）
- 各市场状态一览
- 最近 24 小时的关键指标趋势（转化率、技术失败率）
- 当前活跃告警列表

**健康状态判断规则（待填写）：**
- 健康：所有市场交易失败率 < X%，API 错误率 < Y%
- 警告：任一市场交易失败率 > X%，或 P95 响应时间 > Zms
- 故障：任一市场交易失败率 > X%，持续超过 N 分钟

---

### Module 2：转化漏斗分析（Conversion Funnel）

**主要用户：** PM

**页面功能：**
- 结算漏斗逐步转化率，支持按市场、日期范围筛选
- 每步的流失量和流失率
- 流失原因分布：用户放弃 vs 技术失败（关联 Transaction ID）
- 技术失败的具体原因下钻（api_timeout / sdk_error / ...）

**关键设计点：**
- 漏斗步骤需与后端状态机保持一致
- 同一 transaction_id 出现多次失败，只计一次（去重规则待定）

**指标定义：**
- 步骤转化率 = 完成当前步骤的 transaction 数 / 进入当前步骤的 transaction 数
- 技术失败率 = 因技术原因失败的 transaction 数 / 所有进入 payment_submitted 的 transaction 数

---

### Module 3：支付渠道表现（Payment Channel Performance）

**主要用户：** PM、电商领导

**页面功能：**
- 各支付渠道的成功率，按市场分组
- 渠道用户参与度：被选择率、被选择后的完成率
- 各渠道 P50/P95 支付响应时间
- 渠道间横向对比（支持自定义时间范围）

**指标定义：**
- 渠道选择率 = 选择该渠道的 transaction 数 / 进入 payment_selected 步骤的 transaction 数
- 渠道成功率 = 该渠道支付成功数 / 该渠道发起支付数

---

### Module 4：技术错误追踪（Technical Error Tracking）

**主要用户：** 工程师

**页面功能：**
- API 错误率和响应时间，按接口和市场分组
- 支付 SDK 错误分布（错误码 + 频率）
- 错误影响量化：X 个错误影响了 Y 笔交易，约 €Z
- 错误与 transaction_id 关联，支持单笔追踪

**关键设计点：**
- 需要区分"影响交易的错误"和"不影响交易的错误"
- 同一笔交易的重试不应该重复计入失败

---

### Module 5：支付账号健康（Payment Account Health）

**主要用户：** 工程师、各市场运营负责人

**范围：** 仅做异常检测，不做状态管理

**页面功能：**
- 各市场支付账号的成功率趋势（近 7 天）
- 异常标记：成功率相比过去 7 天均值下降超过 X%
- 告警记录

**数据来源（待确认）：**
- 方案 A：聚合支付 SDK 回调数据（不需要账号权限）
- 方案 B：轮询各支付平台 API（需要各市场配合提供权限）

> 建议优先方案 A，方案 B 作为增强数据源

### 数据来源

| 数据类型 | 来源 | 接入方式 |

| 前端用户行为事件 | 自建埋点（携带 transaction_id） | 直接上报 |

| 支付 SDK 回调 | 前端 SDK 事件 | 埋点上报 |

## 七、扩展性设计

### 新市场上线

- 所有市场相关配置通过配置文件管理，不 hardcode
- 新市场上线只需：添加市场配置 + 告警通知人

### 新支付方式接入

- 支付渠道通过 ID 标识，监控逻辑与具体渠道解耦
- 新渠道接入只需：注册渠道 ID + 确认 SDK 事件携带 transaction_id

### 线下渠道（本版本不做，预留扩展点）

- transaction_id 的生成逻辑需支持 offline 渠道标识
- 漏斗步骤定义需支持不同渠道的不同步骤结构

## 八、不做的事（边界）

- 风险合规层（Layer 5）：v1 不做，后续视财务/法务需求接入
- 线下渠道监控：预留扩展点，v1 不覆盖
- 自定义 dashboard：v1 只提供预定义视图
- 历史数据回填：从系统上线时间开始采集，不回填历史
- 完整的反欺诈系统：不在此系统范围内
